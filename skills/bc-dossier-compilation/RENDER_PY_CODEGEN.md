# `render.py` codegen — how the agent writes and refreshes the dossier script

This document specifies the agent-authored `render.py` flow described in SKILL.md Step 5. It is the contract between the agent's reasoning (which decides *what* goes in the dossier) and the plugin's Python module (which decides *how* it gets rendered).

The codegen module is `scripts/be_civic_dossier/codegen.py`.

## What gets generated

The agent writes a script at:

```
<user-project>/<procedure-id>/dossier/render.py
```

The path follows the `documents/<procedure-id>/` convention so multi-procedure
projects (rare but possible) don't collide. The script lives inside the user's
local project folder, never in the plugin install — the user can edit it, the
agent re-reads it on every refresh, and removing the procedure folder is the
clean way to start over.

## Shape of `render.py`

The codegen module emits a script with **three concerns kept separate** so each
can be edited independently:

1. **Import header** — auto-generated, marked "do not edit by hand", contains
   the absolute-path + `CLAUDE_PLUGIN_ROOT` + `ImportError` probe fallback per
   SKILL.md Step 3. Reasoning is at the bottom of this doc.
2. **`Dossier(...)` constructor** — written once at script creation; preserved
   verbatim on refresh. The user can edit fields (e.g. correct the spelling of
   their name or change the filing authority) and the agent will not stomp on
   their edits.
3. **Agent-managed item list** — bracketed by `# BEGIN agent-managed items
   -- regenerated on refresh` and `# END agent-managed items` marker comments.
   The agent rewrites the contents on every refresh based on what is in the
   `documents/<procedure-id>/` folder and what is still placeholder.

Outside the markers the user can freely add their own `d.add(...)` calls
(e.g. a custom note or an extra optional appendix). The codegen module does
not touch anything outside the markers.

A minimal generated file:

```python
"""Be Civic dossier renderer for nationality-application.

Generated by the be-civic-plugin agent. Edit and re-run to regenerate.
...
"""

from __future__ import annotations

import os
import sys
from pathlib import Path

# ---------------------------------------------------------------------------
# Plugin path injection — auto-generated, do not edit by hand.
# ---------------------------------------------------------------------------
_PLUGIN_ROOT_BAKED = r"/home/<user>/.claude/plugins/be-civic-plugin"

def _resolve_plugin_root() -> Path:
    candidates = []
    env = os.environ.get("CLAUDE_PLUGIN_ROOT")
    if env:
        candidates.append(Path(env).expanduser())
    candidates.append(Path(_PLUGIN_ROOT_BAKED))
    for candidate in candidates:
        if candidate.is_dir() and (candidate / "scripts" / "be_civic_dossier").is_dir():
            return candidate.resolve()
    raise RuntimeError("Could not locate the be-civic-plugin install. ...")

_PLUGIN_ROOT = _resolve_plugin_root()
sys.path.insert(0, str(_PLUGIN_ROOT / "vendor"))
sys.path.insert(0, str(_PLUGIN_ROOT / "scripts"))

try:
    from be_civic_dossier import (
        Dossier, IdCard, FullPageCert, MultiPageDoc, FeeReceipt,
        FilledForm, Placeholder,
    )
except ImportError as exc:
    raise RuntimeError(
        f"Failed to import be_civic_dossier from {_PLUGIN_ROOT}: {exc}. "
        "Re-run the dossier compilation skill so the agent regenerates this "
        "file's import header with a working plugin path."
    ) from exc

# ---------------------------------------------------------------------------
# Dossier definition — edit freely.
# ---------------------------------------------------------------------------
d = Dossier(
    procedure_id="nationality-application",
    procedure_title="Belgian nationality declaration (art. 12bis §1, 2°)",
    conversation_language="en",
    filing_language="fr",
    applicant_name="Henry Kernot",
    filing_authority="Commune d'Ixelles — Service Population, Nationalités",
    filing_date="2026-05-19",
)

# BEGIN agent-managed items -- regenerated on refresh
d.add(FullPageCert(
    "../documents/birth-certificate-apostilled.pdf",
    cert_type="Birth certificate (apostilled)",
    original_required=True,
))
d.add(Placeholder(
    cert_type="Annexe 1 — Déclaration de nationalité (signed at commune)",
    source="commune visit",
))
# END agent-managed items

officer_notes_path = Path(__file__).parent / "officer-notes.md"
if officer_notes_path.is_file():
    d.add_officer_notes(officer_notes_path.read_text(encoding="utf-8"))

if __name__ == "__main__":
    out = d.render(output_path="dossier-2026-05-19.pdf")
    print(f"Wrote {out}")
```

## API summary

```python
from be_civic_dossier import (
    DossierConfig,
    render_render_py,
    refresh_items,
    parse_existing_items,
    resolve_template, load_template,
)
```

### `render_render_py(config: DossierConfig, plugin_root_abs: Path | None = None) -> str`

Return the full text of a fresh `render.py`. The script is **deterministic**: same
inputs produce the same bytes (the filename in the call to `d.render(...)` is
the only non-deterministic field, and that's controlled by `output_filename`).

`plugin_root_abs` overrides the auto-detected plugin path — used when generating
from a context that isn't inside the plugin checkout (e.g. when the plugin is
installed under `~/.claude/plugins/`).

### `refresh_items(existing_source: str, new_items_source: str) -> str`

Rewrite the agent-managed item-list block inside an existing `render.py`. Raises
`ValueError` if the markers are missing — that's the "user rewrote the file by
hand" guard. Caller can decide to (a) fall back to full regeneration or (b)
prompt the user before clobbering.

### `parse_existing_items(source: str) -> Iterable[ParsedItem]`

Best-effort AST walk that yields each `d.add(SomeClass(...))` as a
`ParsedItem(class_name, args, kwargs)`. Useful for the reconcile step where the
agent compares what the user currently has against what the procedure checklist
demands. Advisory only — don't drive destructive edits off this alone.

### `resolve_template(name, project_templates_dir=None) -> Path` / `load_template(...)`

Resolve a template by canonical name. The codegen module prefers a project-local
override at `<project>/<procedure-id>/dossier/templates/<name>` before falling
back to the plugin-bundled copy. Used by the renderer when it loads
templates at render time.

## Refresh flow — preserve user edits

Per SKILL.md Step 5: "Write or update `render.py` in the user's project folder
— preserve user edits, only modify the item list." The codegen module enforces
this with marker comments and `refresh_items()`. The agent's contract:

1. On **first** dossier event for a procedure, call `render_render_py(config)`
   and write the result to `<project>/<procedure-id>/dossier/render.py`.
2. On every **subsequent** dossier event:
   1. Read the existing file.
   2. Call `parse_existing_items(source)` to see what's there.
   3. Build a fresh items-source string based on the current state of
      `documents/<procedure-id>/` (and what the checklist still expects).
   4. Call `refresh_items(existing, new_items_source)`.
   5. Write the result back.
3. If `refresh_items` raises `ValueError`, the file has been hand-edited
   outside the markers. **Surface this to the user in chat**: "Your render.py
   looks hand-edited — want me to regenerate from scratch, or leave it?"
   Don't silently rewrite.

## Why the absolute-path + env-var + ImportError-probe header

The header handles the case where the plugin moves on disk but an old `render.py` is still around. It tries three locations in order:

1. **`CLAUDE_PLUGIN_ROOT` env var** — set by the plugin harness when it's
   driving the script. If the user re-runs `render.py` from inside a plugin
   session, this is correct by construction.
2. **Baked-in absolute path** (the `_PLUGIN_ROOT_BAKED` literal) — captured at
   codegen time. Works for any direct `python3 render.py` invocation as long
   as the plugin still lives where it was installed.
3. **`ImportError` raise with self-healing instruction** — if neither path
   resolves a real plugin install, the script raises with an explicit message
   telling the agent to re-run `bc-dossier-compilation` so the header can be
   regenerated against the current plugin location.

This is deliberately verbose: the dossier is a procedure-state artefact, not
throwaway code. The user might re-run an old `render.py` weeks later after
upgrading the plugin, and we'd rather raise a clear error than fail with a
cryptic `ImportError`.

## Coordination with the renderer module

The renderer module owns `Dossier`, `IdCard`, `FullPageCert`, `MultiPageDoc`,
`FeeReceipt`, `FilledForm`, `Placeholder`, and `watermark`. They live in
sibling modules inside `scripts/be_civic_dossier/`. The `__init__.py` re-exports
them defensively (`try / except ImportError` → `None`) so the codegen path
remains usable even when the renderer modules are not yet available — the
codegen is a pure-Python concern with no runtime dependency on the renderer.

The contract is that the public class names are exported from the package;
the internal module layout is the renderer's concern.

## Out of scope

- **Validating that field values match what the receiving authority expects.**
  The agent does that before writing the script; this module is structural.
- **Filling Annexe 1 from `profile.json`.** Filling values is the renderer's
  job. This module emits the `FilledForm(...)` call; the renderer reads the
  user's `profile.json` and the procedure's filled-fields spec.
- **Watermarking.** Applied at render time by `be_civic_dossier.watermark`.
  The templates carry the structure; the watermark is overlaid.
